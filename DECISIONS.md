# Decision Log

| Date | Decision | Options Considered | Chosen | Rationale |
|------|----------|-------------------|--------|-----------|
| 2026-03-24 | GHL opportunities search HTTP method | GET with query params vs POST with body | GET with snake_case query params (`location_id`, `pipeline_id`, `limit`) | POST method ignores `pipeline_id` param entirely — returns unfiltered results from all pipelines. GET properly filters. |
| 2026-03-24 | Disposition deals stage filter | Show all stages vs Marketing Active and beyond | Marketing Active and beyond (6 stages) | Bobby specified only active disposition properties should appear. Excludes Intake/Prep, Closed – Paid, Dead / Withdrawn. |
| 2026-03-24 | CRM Tracker top stats cards | Under Contract + Total Contacts vs Weekly WoW metrics | New Acq Opps (7d) + Buyer Leads (7d) with week-over-week | More actionable than static counts. Under Contract still visible in deals section below. |
| 2026-03-24 | GHL error handling verbosity | Silent throw vs capture response body | Capture response body in error message | Bare status codes made debugging GHL API issues difficult. Including the response body surfaces the actual validation error. |
| 2026-03-24 | Session log format and location | Single SESSION_LOG.md vs individual files in `sessions/` | Individual files in `sessions/` | Easier to scan by date, avoids one massive file, better for context handoff between sessions. |
| 2026-03-25 | Vercel env var method | `echo` pipe vs `printf '%s'` pipe | `printf '%s'` (no trailing newline) | `echo` appends `\n` that Vercel stores literally, corrupting secrets. Zoom OAuth rejected `client_id\n`. All 9 env vars were affected. Now a mandatory protocol for all Vercel deployments. |
| 2026-03-25 | Zoom meeting idempotency key | `recording.id` (series ID) vs `recording.uuid` (instance ID) | `recording.uuid` | Recurring meetings share the same `id` across all instances. Only the first instance was ever ingested; all subsequent ones were skipped as duplicates. `uuid` is unique per meeting occurrence. |
| 2026-03-25 | Defensive array checks on Claude extraction | Trust tool response vs guard with `Array.isArray()` | Guard with `Array.isArray()` | Claude occasionally returns `null` instead of an empty array for `discussion_topics` or `action_items`, crashing the ingest pipeline. Defensive check converts to empty array. |
| 2026-03-30 | Vercel cron HTTP method | POST handler vs GET handler | GET handler | Vercel's cron scheduler sends GET requests. Both cron routes only exported POST, so scheduled jobs silently returned 405 and never ran. All Vercel cron routes must export GET. |
| 2026-04-06 | Buyer blast orchestration engine | Vercel serverless vs Trigger.dev | Trigger.dev | Drip-send blasts are long-running (minutes to hours with delays between sends). Vercel serverless has 10s/60s timeout limits. Trigger.dev provides durable execution with retries. |
| 2026-04-06 | SMS delivery API | GHL direct SMS API vs GHL conversations API | GHL conversations API | Conversations API handles thread context and enables delivery status tracking via webhooks. |
| 2026-04-06 | Blast test mode strategy | Separate test endpoint vs tag-based filter | Tag-based filter ("test" tag) | Simpler — same code path, same webhook, just filters to contacts tagged "test". Remove tag check when ready for production. |
| 2026-04-06 | CRM sync cron frequency | Every 6 hours vs daily | Daily | Vercel Hobby plan limits to 2 cron jobs. Daily frequency is sufficient for CRM data freshness. |
| 2026-04-06 | Blast state storage | In-memory vs Supabase | Supabase (blast_runs, blast_recipients, sending_tier tables) | Consistent with existing project storage. Enables blast history, recipient tracking, and tier management across restarts. |
| 2026-04-07 | GHL custom field value types | Trust GHL types vs coerce to string | Always coerce with `String()` before `.trim()` | GHL returns custom field values as numbers, arrays, or strings inconsistently. Never trust the type — always coerce. |
| 2026-04-07 | GHL task creation required fields | Minimal payload vs include dueDate + completed | Include `dueDate` (7 days out) and `completed: false` | GHL returns 422 without these fields despite them not being prominently documented. Undocumented required fields. |
| 2026-04-07 | Buyer state matching | Abbreviated + full state fallback vs abbreviated only | Abbreviated state only | Full state name is a data quality issue. Standardize at source — every opportunity must have abbreviated state populated. |
| 2026-04-07 | GHL dndSettings data shape | Assume array vs handle as object keyed by channel | Object keyed by channel (e.g., `{ SMS: { status: "active" } }`) | Live API response confirmed it's an object, not an array. Buyer filter was excluding everyone due to wrong shape assumption. |
| 2026-04-07 | Buyer contact fetching strategy | Paginate all contacts (client-side filter) vs GHL search endpoint (server-side filter) | GHL search endpoint with server-side tag filtering | Paginating 3,400+ contacts required 34 API calls. Search endpoint returns only matching contacts — 3 results in 1 call. |
| 2026-04-07 | Blast idempotency key scope | Opportunity ID only vs opportunity ID + timestamp | Opportunity ID + timestamp | Opportunity-only key blocked intentional re-blasts after bug fixes. Business-level duplicate prevention belongs in task logic (`findExistingBlast`), not the trigger idempotency key. |
| 2026-04-07 | GHL SMS recipient identifier | conversationId vs contactId | contactId | GHL's `POST /conversations/messages` requires `contactId`. No need to search/create conversations first — send directly with contactId. |
